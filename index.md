<h2>Professional Self-Assessment</h2>

<p>When I began my educational journey at SNHU, I was focused almost entirely on data analysis. That started to change as I worked through the Computer Science program and was exposed to a wider range of concepts and projects. I began to see that there was more to the field than just analyzing data, especially when it came to building and improving systems. Over time, my focus shifted toward understanding how different areas of computer science connect and how they come together when working on real problems. The sections that follow break down the core course outcomes and my capstone work, showing how these experiences have shaped my skills and prepared me for the field.</p>

<hr style="border: none; height: 1px; background: #374151; margin: 28px 0;">

<h3>Collaboration in a Team Environment</h3>

<p>Although I had some experience with collaboration earlier in my program, the course that really gave me practical, hands-on teamwork with other students and my professor was CS-310: Collaboration and Team Project. One of the biggest things I took away from this course was how important version control is when multiple people are working in the same codebase. I used Git and Bitbucket to manage my changes, which forced me to stay organized and think about how my work would affect the rest of the team. At times, I had to deal with merge conflicts and overlapping changes, which made it clear pretty quickly why a structured workflow matters. Communication also played a big role. If something wasn’t clear or coordinated, it could cause issues for everyone else. By the end of the course, I had a much better understanding of how collaboration actually works in practice and why it’s such an important part of software development.</p>

<hr style="border: none; height: 1px; background: #374151; margin: 28px 0;">

<h3>Communicating with Stakeholders</h3>

<p>Throughout my time at SNHU, I had several projects that involved communicating with different stakeholder groups. CS-255: System Analysis and Design was one of the earlier courses that really jumpstarted my awareness of how important that is. In that course, I worked on building a System Design Document, and it ended up being more involved than I expected. I had to derive requirements from non-technical stakeholders who could describe what they needed, but not in a structured or technical way. That meant breaking things down and figuring out what they were actually asking for. From there, I defined both functional requirements, what the system should do, and non-functional requirements, how it should perform. That process alone changed how I think about building systems. I also worked with UML diagrams like use case and activity diagrams. Those helped me communicate how the system flows and how different parts interact without over-explaining everything. It made a big difference. It showed me that communication is not just about explaining your work. It is a two-way process, and for a product to be fully successful, clear communication between all stakeholders is essential.</p>

<hr style="border: none; height: 1px; background: #374151; margin: 28px 0;">

<h3>Data Structures and Algorithms</h3>

<p>DAT-430: Leveraging Data Analysis for Organizational Results was, for me personally, one of the most challenging and rewarding courses I took at SNHU, and probably my favorite. It was one of the first times I worked with real datasets in a way that went beyond basic analysis. I had to think about how the data was structured and how to prepare it before progressing any further in the project. Data cleaning and manipulation took up a lot of that time. That process taught me pretty quickly that if the data is not clean and consistent, the entire project starts to fall apart. I also worked on feature engineering to create variables that actually revealed patterns. That is where I started working directly with algorithms, using models like logistic regression, decision trees, and Random Forest and seeing how different approaches revealed key variables actually influencing attrition.</p>

<p>Once I had algorithmic proof of which variables actually mattered, I started crafting visualizations to highlight those trends and support the attrition predictions. That led to building my HR Attrition Dashboard and getting hands-on experience with Tableau. The course also pushed me to present my findings, so I created a PowerPoint presentation with speaker notes that clearly explained how the company could use the data to reduce turnover and improve productivity. This course truly helped expand my understanding of data structures and algorithms by showing how they are applied in real-world data analysis and decision-making, and it felt like a real-world project I would actually be working on in the data analysis field.</p>

<hr style="border: none; height: 1px; background: #374151; margin: 28px 0;">

<h3>Software Engineering and Databases</h3>

<p>At one point in the program, I started moving away from working on smaller pieces of code and toward building something that actually felt like a full application. CS-340: Client/Server Development was a standout course for me that brought the database, the code layer, and the front-end UI together to work in tandem as one system. I got hands-on experience with MongoDB on the backend, importing the animal shelter CSV dataset and working directly with it in the database. I also improved CRUD operations so the data could be created, read, updated, and deleted in a more structured way, and handled role-based database access to control how different users interact with the system. I then used PyMongo in the middle layer to pull and aggregate the data and connect it to a Dash application. That allowed me to create an interface with an animal table, a geolocation chart, and radio filter buttons so users could quickly narrow down animals for rescue training. This experience helped me better understand how software and databases actually work together in a real application.</p>

<hr style="border: none; height: 1px; background: #374151; margin: 28px 0;">

<h3>Security</h3>

<p>Early on in my SNHU education, I treated software security as something that was important, but not always a top priority. That changed once I took CS-305: Software Security. The course taught me not only the different ways software breaches and attacks can happen, but also how to actually address them. In the Global Rain project, I worked on securing a financial services web application that handled sensitive client data. I added SHA-256 hashing to protect that data and implemented SSL certificates so the application could run over secure HTTPS communication. I spent time identifying vulnerabilities in the system, especially those tied to outdated dependencies. That part showed me pretty quickly how risky outdated libraries can be and how important it is to keep them updated. I also addressed potential injection risks and added proper error handling so sensitive information would not be exposed. This experience changed how I approach development, and security is now something I consider crucial in every part of an application.</p>

<h2>Capstone Reflection</h2>

<p>The artifacts in this portfolio are all based on my CS-340 Grazioso Salvare project. I chose it because it already worked as a full client/server application and gave me something real to build on. The original version included a MongoDB backend, basic CRUD operations, PyMongo aggregation, and a Dash interface that allowed Grazioso employees to filter and visualize animal shelter data. From there, I focused on improving three main areas. For software engineering, I moved the project out of Jupyter and into Visual Studio Code and reorganized it into a modular application. Separating the database logic, application logic, and UI made the system easier to manage and easier to expand. I also added an admin login system with password hashing and full CRUD functionality through the interface. On the database side, I pushed more of the data processing into MongoDB using aggregation pipelines so the app could work with cleaner, structured data. For algorithms and data structures, I moved beyond basic filtering and used grouping and categorization to support more meaningful analysis.</p>

<p>These improvements all build on each other. As the data became more structured and the queries more efficient, the application was able to support more advanced features. I added an analytics page with shelter scorecard metrics, breed distribution charts, a breed outcome sunburst, and a trend chart over time. Those visuals rely on the updated aggregation logic and improved data structuring. The result is a system that runs more efficiently and gives clearer insight into the data. I also worked on security, testing, and code quality through better validation and more consistent commenting. Each artifact in this portfolio has its own narrative page that explains the enhancements in more detail. Together, they show how I can take an existing application, improve how it is structured, and turn it into something more useful for supporting real decision-making, which is directly in line with my goals in computer science and data analysis.</p>
